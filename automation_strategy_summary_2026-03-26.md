# 定时执行方案评估总结（news-monitor）

日期：2026-03-26
项目：Academic Research Monitor / news-monitor

## 一、结论

**对这个项目，最划算的方案是：**

> **由你自己的程序负责“定时调度 + 抓取 + 去重 + 重试 + 出错告警 + 落盘”，只在需要自然语言理解/摘要/分类的步骤调用 AI。**

也就是说：

- **不要**把 Codex / Claude Code 当成主定时器和主控制器；
- **要**把它们当成流程中的“智能处理节点”。

---

## 二、为什么这个方案更合适

### 1. 从稳定性看：程序定时更可靠

对 `news-monitor` 这类任务，核心诉求通常是：

- 固定时间执行；
- 抓取失败可重试；
- 某个源失败不能拖垮整批任务；
- 输出文件要稳定生成；
- 最好有日志、状态、告警。

这些都更适合由**确定性程序**处理，而不是交给一个面向交互的 coding agent 来充当总控。

你当前项目本身已经具备这条路线的雏形：

- 有 `run.py` 作为主流程；
- 有 `crontab`；
- 有 Docker / `docker-compose.yml`；
- 有 `output/` 与 `logs/`；
- 配置与邮件、报告、LLM provider 已经拆分。

这说明：**你现在的代码结构天然更适合“程序定时执行”**，而不是“让 agent 定时接管整条链路”。

### 2. 从费用看：程序主控通常更省

如果把 Codex / Claude Code 当“定时代理平台”，你往往会产生这些额外消耗：

- 每次任务都要重新进入 agent 上下文；
- 为了让 agent 理解目标，需要重复注入 instructions / skill / repo context；
- agent 可能会执行比你写死脚本更多的思考步骤；
- 成本波动更大，也更难做精细预算。

而如果由程序主控：

- 非 AI 步骤（抓取、过滤、去重、模板渲染、邮件发送）几乎是固定成本；
- 只有“摘要 / 归纳 / 趋势判断”才会消耗模型；
- 后续可以灵活切换模型：高质量模型用于最终报告，便宜模型用于初筛；
- 更容易做 token 限流、批处理、缓存与降级。

**所以长期看，程序主控 + 按需调用 AI，一般更便宜。**

## 三、三种方案对比

### 方案 A：你自己写程序定时执行（推荐）

**结构：**
`cron / systemd / GitHub Actions / 云函数` → `run.py` → 抓取/去重/分析/生成报告/发送邮件

**优点：**
- 最稳定；
- 最容易加重试、超时、幂等、告警；
- 成本最好控；
- 能做完整日志与审计；
- 更适合生产任务。

**缺点：**
- 你要自己维护调度、部署和异常处理；
- 初期工程化工作略多。

**适用性：**
- 非常适合当前 `news-monitor`。

---

### 方案 B：让 Codex 定时执行 skill / instructions

**优点：**
- 对“日报生成、总结、review queue、代码审查类”任务很方便；
- 自动化体验好，适合半自动工作流。

**缺点：**
- 更适合“允许审阅的人机协作型任务”，不一定适合作为严格生产调度器；
- 对抓取、重试、精细错误处理、文件状态管理，不如你自己的程序确定；
- 成本通常比纯脚本更难预测。

**适用性：**
- 可作为补充层，例如：每天自动读结果并生成“面向人的总结”；
- 不建议作为底层主调度器。

---

### 方案 C：让 Claude Code 定时执行 skill / 命令

**优点：**
- 对开发、重构、交互式分析很强；
- 适合“我想让 agent 帮我写/改这套系统”。

**缺点：**
- 若用于定时生产执行，通常仍然需要外部 scheduler 拉起；
- 更像开发代理，不像稳定任务编排器；
- 成本与执行路径波动通常也较大。

**适用性：**
- 适合开发阶段；
- 不建议直接担任长期生产调度主角。

---

## 四、对当前项目的最佳落地方式

### 推荐架构

#### 第 1 层：确定性执行层（你自己的程序）
负责：
- 定时触发；
- 抓取源站；
- 超时控制；
- 去重；
- 失败重试；
- 生成 HTML/PDF；
- 发送邮件；
- 写日志；
- 可选：保存历史结果到 SQLite / JSON。

#### 第 2 层：AI 能力层
负责：
- 摘要；
- 主题归类；
- 创新点提取；
- 趋势总结；
- 最终中文报告润色。

#### 第 3 层：可选 agent 协作层
只在这些场景使用 Codex / Claude Code：
- 帮你改代码；
- 帮你调 prompt；
- 帮你分析失败日志；
- 帮你生成更高质量的“管理层摘要”；
- 帮你定期审查系统质量，而不是代替系统运行。

## 五、最现实的成本判断

### 如果你的目标是“每天稳定出报告”
最省钱的通常不是“全程 agent 化”，而是：

1. 先用脚本把 80% 的确定性工作做掉；
2. 再把 20% 真的需要语言理解的部分交给模型；
3. 必要时做分层模型：
   - 便宜模型：预筛、标签、粗摘要；
   - 贵模型：最后趋势总结、精修中文版报告。

### 什么情况下 agent 化值得
只有当你更在意这些价值时，才更值得让 Codex / Claude Code 参与自动化：

- 希望少写工程代码；
- 接受成本波动；
- 接受结果需要人工 review；
- 任务本身更偏“开放式总结”，而不是“严格执行流水线”。

对 `news-monitor` 而言，这些不是主诉求。

---

## 六、最终建议

### 推荐决策

**主方案：继续走“程序定时执行”路线。**

具体就是：

- 保留 `run.py` 作为唯一入口；
- 保留 `crontab` / Docker 定时；
- 把抓取、分析、报告、邮件都放在程序编排里；
- Codex / Claude Code 只作为“开发助手”或“报告增强器”。

### 一句话版

> **如果你要的是生产级稳定性和长期低成本：自己写程序定时跑。**  
> **如果你要的是开发效率和临时自动化体验：让 Codex / Claude Code 参与，但不要让它们做底层调度主控。**

---

## 七、建议的下一步

按优先级建议你这样做：

1. 继续使用现有 `cron + run.py + Docker` 方案；
2. 增加运行状态落盘（例如 `output/run_status.json`）；
3. 增加失败重试与告警；
4. 给 LLM 分析加缓存，避免重复花费；
5. 如需提升可读性，再额外加一个“AI 总结层”；
6. 若以后要做多任务编排，再考虑把调度迁移到 GitHub Actions、云函数或专门工作流系统。

---

## 八、补充说明

本结论是基于两类判断：

1. **项目本地结构**：当前仓库已经明显偏向“程序主控”的设计；
2. **产品能力与价格维度**：agent 自动化更适合协作/总结型任务，而程序调度更适合生产流水线。

因此，从**稳定可靠性**与**总体费用可控性**两个维度综合看，**程序定时执行更划算**。
